# Harbor 实现路线图 v0.1

## 1. 文档目标

本文档用于定义 Harbor 从当前阶段到 MVP 可运行版本的实际实现路线。它不是团队分工文档，而是项目的执行顺序说明，用于指导后续的设计、开发、联调和验证。

当前路线图以如下前提为基础：

- Harbor 第一阶段采用桌面客户端形态
- 客户端需要支持本地工作目录接入
- 当前阶段直接复用 `services/deer-flow` 作为后端 Agent Runtime
- 登录与鉴权不在本阶段处理，后续再在 DeerFlow 前补 Harbor 自己的 Auth / BFF 层
- 当前目标是先做出一个可运行的 MVP，而不是一次性实现完整平台能力

## 2. 总体原则

实现顺序遵循以下原则：

- 先确定边界，再开始编码
- 先搭骨架，再接能力
- 先跑通主链路，再做体验优化
- 先实现目录上下文接入，再讨论复杂本地任务执行
- 不提前实现后续才需要的复杂能力

## 3. 阶段划分

Harbor 第一阶段建议分为 6 个阶段推进。

### 阶段 1：设计定稿

目标：把第一版产品范围和架构边界定清楚。

当前阶段需要完成：

- MVP 定义文档
- 系统架构设计文档
- 客户端架构设计文档
- 实现路线图文档

完成标准：

- 明确第一版做什么、不做什么
- 明确客户端、服务端、Agent 层、模型层的职责边界
- 明确客户端为何采用桌面应用

当前状态：

- 已基本完成

### 阶段 2：客户端骨架搭建

目标：搭建 Harbor 桌面客户端基础工程，并做出可交互的产品壳。

本阶段需要实现：

- Electron + React + TypeScript + Vite 工程初始化
- 基础目录结构
- 主窗口创建
- 左侧会话栏 UI
- 主聊天区 UI
- 底部输入区 UI
- 基础深色主题和设计 token
- 本地目录选择能力接入
- 当前工作目录展示

本阶段暂不依赖真实服务端，可以先使用本地 mock 数据。

完成标准：

- 客户端能够启动并显示完整主界面
- 用户可以选择本地目录
- 用户可以在界面中看到当前工作目录状态
- 会话列表和消息区具备静态交互能力

交付物：

- 客户端可运行基础工程
- 第一版桌面 UI

### 阶段 3：DeerFlow 联调准备

目标：把 DeerFlow 作为现成后端跑起来，并明确 Harbor 的接入边界。

本阶段需要实现：

- 确认 DeerFlow 本地启动方式
- 确认 Gateway / LangGraph 对外地址
- 确认 Harbor 当前需要使用的 API
- 明确本地工作目录当前只能作为上下文提示，不能直接视为 DeerFlow 可访问路径
- 写清楚后续鉴权层将放在 DeerFlow 前面，而不是直接写进 DeerFlow

本阶段暂不引入 Harbor 自己的新后端工程。

完成标准：

- DeerFlow 能在本地稳定启动
- Harbor 已明确可接入的最小接口集合
- 文档已从旧方案切换到“先直连 DeerFlow”的新方案

交付物：

- DeerFlow 接入方案文档
- 后续实施边界说明

### 阶段 4：聊天链路打通

目标：打通客户端 -> DeerFlow 的核心对话链路。

本阶段需要实现：

- Harbor 调 DeerFlow 健康检查
- Harbor 调 DeerFlow `/api/runs/wait` 或 `/api/runs/stream`
- thread_id 保持
- 顶部连接状态展示
- 基础错误处理和超时处理

完成标准：

- 用户可以在客户端发送消息并获得真实 Agent 回复
- 当前会话能复用同一个 DeerFlow thread
- 客户端里不再是本地 mock 回复

交付物：

- 可用的真实聊天链路

### 阶段 5：MVP 能力补全

目标：补足第一版定义中的核心体验能力。

本阶段需要实现：

- 文件与图片上传接入实际聊天链路
- 历史会话切换与恢复
- 客户端连接状态展示
- 基础错误提示
- 附件展示卡片
- 空状态和加载状态
- 基础设置页或连接配置入口

完成标准：

- 用户可以完成一轮完整的真实使用流程
- 系统具备基础可用性和可演示性

交付物：

- Harbor MVP 可运行版本

### 阶段 6：MVP 打磨与验证

目标：让 MVP 从“能跑”提升到“能稳定试用”。

本阶段需要实现：

- UI 细节打磨
- 日志和异常排查
- 局域网连接验证
- 文件上传边界验证
- 多轮会话验证
- 基础性能检查
- 打包与发布流程验证

完成标准：

- 团队内部可试用
- 主要链路稳定
- 具备最基本的演示和迭代能力

交付物：

- MVP 试用版本

## 4. 当前建议的实际起点

如果从现在正式进入实现，我建议立刻开始：

**阶段 3：DeerFlow 联调准备** 与 **阶段 4：聊天链路打通**

原因如下：

- 客户端骨架和基础 UI 已经完成
- 当前最大的阻塞已经不是界面，而是真实后端没有接进去
- DeerFlow 已经在仓库内，可以直接拿来验证真实链路
- 先把真实链路跑通，再谈登录层和 Harbor 自己的后端更稳

也就是说，接下来最先动手的是：

- 校正文档到 DeerFlow 路线
- 用 Harbor 直接调用 DeerFlow
- 把 mock 回复替换成真实回复

## 5. 当前阶段后的执行顺序

建议的实际开发顺序如下：

1. 校正 Harbor 文档中的旧架构表述
2. 确认 DeerFlow 本地运行方式和最小接口集合
3. 让桌面端接入 DeerFlow 健康检查
4. 把聊天 mock 替换成 DeerFlow 真实回复
5. 保持 thread_id，实现多轮会话基础上下文
6. 补连接状态、错误状态和空状态
7. 接文件上传与 artifact 展示
8. 接历史会话和线程恢复
9. 视需要补 Harbor Auth / BFF
10. 做 MVP 打磨和验证

## 6. 当前不建议立即做的事情

为了避免偏离 MVP 路线，当前不建议立即投入以下内容：

- 复杂 Skill 系统
- 自动流水线执行
- UE Trace 专项解析
- 插件市场或插件框架
- 多模型调度系统
- 完整权限体系
- 大规模部署和集群化改造

这些都应该放到 MVP 跑通之后。

## 7. 对后续实现的要求

后续实现时，应始终遵守以下要求：

- 代码结构服务于后续扩展
- 配置不要写死
- 目录上下文能力先做最小可用版本
- UI 先追求稳和清晰，再追求复杂特效
- 所有实现都围绕主链路展开

## 8. 一句话总结

Harbor 从现在开始的正确推进方式是：先把桌面端直接接上 DeerFlow，把真实聊天链路跑通；之后再补上传、历史会话和状态展示，最后再在 DeerFlow 前面加 Harbor 自己的登录鉴权层。
