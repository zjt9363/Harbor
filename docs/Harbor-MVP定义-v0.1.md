# Harbor MVP 定义 v0.1

## 1. 项目目标

Harbor 是一个面向团队内部使用的本地化 Agent 平台。第一阶段目标是搭建一个桌面客户端，让团队成员能够通过统一聊天界面与本地部署的开源 Agent 交互，并逐步接入本地工作目录、历史会话和附件能力。

## 2. 目标用户

第一版面向团队内部成员使用，包括但不限于：

- 客户端开发
- 工具开发
- 技术支持人员
- 后续可能接入性能分析或验证流程的工程成员

MVP 阶段默认用户规模较小，以内部试用和验证为主。

## 3. MVP 要解决的问题

第一版只解决最基础的问题：

- 团队成员可以通过桌面客户端访问统一入口
- 用户可以在客户端中与本地 Agent 对话
- 用户可以上传文本文件和图片作为对话输入
- Agent 可以返回文本回复，并在客户端中展示
- 系统可以保存基础的历史会话和消息记录

也就是说，MVP 的目标不是“做一个完整智能平台”，而是先跑通一条稳定的使用主链路。

## 4. MVP 范围

第一版包含以下能力：

- 桌面客户端聊天界面
- 用户输入文本并发送消息
- 上传文件和图片
- 展示 Agent 回复
- 保存历史会话
- 选择本地工作目录并展示当前上下文
- 接入一个本地部署的开源模型
- 通过 DeerFlow 转发请求并生成回复

## 5. MVP 明确不做的内容

为了保证项目可以快速落地，以下内容不纳入第一版范围：

- 不做复杂多 Agent 协作
- 不做完整技能编排系统
- 不做自动流水线执行
- 不做 UE Trace 专项解析能力
- 不做复杂权限体系和多租户设计
- 不做高级知识库 / RAG 系统
- 不做移动端客户端
- 不做高度定制的 Prompt 调优体系
- 不做复杂任务调度与异步作业系统

## 6. 系统主链路

第一版主链路定义为：

1. 用户打开 Harbor 桌面客户端
2. 用户输入文本，或上传文件 / 图片
3. 客户端当前直接将请求发送到 DeerFlow
4. DeerFlow 调用 Agent Runtime
5. Agent Runtime 调用模型服务
6. 模型生成回复并返回
7. 客户端展示结果
8. 当前线程上下文、消息和附件能力由 DeerFlow 承接

这条链路必须优先跑通，并保持稳定。

## 7. 第一版推荐架构

MVP 阶段采用如下基础分层架构：

- Harbor 桌面客户端（Electron + React）
- DeerFlow 网关 / LangGraph
- 模型提供方或本地模型服务（如 vLLM / Ollama）
- 后续再补 Harbor 鉴权 / 业务后端代理层

架构原则是：

- 先直连 DeerFlow 跑通主链路
- 模型层与业务层解耦
- 后续可扩展鉴权、技能、任务和分析能力

## 8. 成功标准

当满足以下条件时，可认为 MVP 达标：

- 团队成员可以启动并使用桌面客户端
- 用户可以成功创建会话并发送消息
- 用户可以上传至少一种文本文件和一种图片文件
- Agent 能稳定返回可展示的文本结果
- 历史会话和消息能被保存并再次查看
- 系统在小范围多人使用下可稳定运行

## 9. 一句话总结

Harbor MVP 的目标不是一开始做成完整的智能工程平台，而是先做出一个团队可使用、可聊天、可上传文件、可接 DeerFlow 的基础桌面客户端。
