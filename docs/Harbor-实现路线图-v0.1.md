# Harbor 实现路线图 v0.1

## 1. 文档目标

本文档用于定义 Harbor 从当前阶段到 MVP 可运行版本的实际实现路线。它不是团队分工文档，而是项目的执行顺序说明，用于指导后续的设计、开发、联调和验证。

当前路线图基于以下前提：

- Harbor 第一阶段采用桌面客户端形态
- 客户端需要支持本地工作目录接入
- 当前阶段直接复用 `services/deer-flow` 作为后端 Agent Runtime
- 登录与鉴权不在本阶段处理，后续再在 DeerFlow 前补 Harbor 自己的鉴权 / 业务后端代理层
- 当前目标是先做出一个可运行的 MVP，而不是一次性实现完整平台能力

## 2. 总体原则

实现顺序遵循以下原则：

- 先确定边界，再开始编码
- 先跑通主链路，再做体验优化
- 先做最小可用能力，再做扩展
- 不提前实现后续才需要的复杂能力

## 3. 阶段划分

### 阶段 1：设计定稿

目标：把第一版产品范围和架构边界定清楚。

需要完成：

- MVP 定义文档
- 系统架构设计文档
- 客户端架构设计文档
- 接入方案文档
- 技术选型文档

### 阶段 2：桌面端骨架搭建

目标：搭建 Harbor 桌面客户端基础工程，并做出可交互的产品外壳。

已完成内容包括：

- Electron + React + TypeScript + Vite 工程初始化
- 基础目录结构
- 主窗口创建
- 左侧会话栏
- 主聊天区
- 底部输入区
- 本地目录选择能力
- 日志系统
- 配置中心

### 阶段 3：DeerFlow 联调准备

目标：把 DeerFlow 作为现成后端跑起来，并明确 Harbor 的接入边界。

已完成或已明确的内容：

- 确认 DeerFlow 本地启动方式
- 确认网关与 LangGraph 对外地址
- 确认 Harbor 当前需要使用的 API
- 明确本地工作目录当前只能作为上下文提示
- 写清楚后续鉴权层将放在 DeerFlow 前面，而不是直接写进 DeerFlow

### 阶段 4：聊天链路打通

目标：打通客户端到 DeerFlow 的核心对话链路。

已完成内容包括：

- Harbor 调 DeerFlow 健康检查
- Harbor 调 DeerFlow `/api/runs/wait` 或 `/api/runs/stream`
- `thread_id` 保持
- 顶部连接状态展示
- 基础错误处理和超时处理
- 本地配置中心
- 本地日志系统

完成标准：

- 用户可以在客户端发送消息并获得真实 Agent 回复
- 当前会话能复用同一个 DeerFlow thread
- 客户端里不再是本地模拟回复

### 阶段 5：MVP 能力补全

目标：补足第一版定义中的核心体验能力。

本阶段需要实现：

- 文件与图片上传接入实际聊天链路
- 历史会话切换与恢复
- 更完整的连接状态展示
- 基础错误提示
- 附件展示卡片
- 空状态和加载状态

### 阶段 6：MVP 打磨与验证

目标：让 MVP 从“能跑”提升到“能稳定试用”。

本阶段需要实现：

- 界面细节打磨
- 日志和异常排查
- 局域网连接验证
- 文件上传边界验证
- 多轮会话验证
- 基础性能检查
- 打包与发布流程验证

### 阶段 7：鉴权层补入

目标：在 DeerFlow 前补 Harbor 自己的登录鉴权层。

本阶段主要处理：

- 账号登录
- 用户鉴权
- 权限控制
- 审计日志
- 用户与 DeerFlow thread 的映射

## 4. 当前建议的实际起点

如果从现在正式进入实现，建议立刻开始：

- 阶段 5：MVP 能力补全
- 本地模型部署与 DeerFlow 模型配置

原因如下：

- 客户端骨架和基础界面已经完成
- DeerFlow 直连主链路已经基本打通
- 当前最大的阻塞已经从“界面与链路”转为“模型部署与可用性”
- 现在最值得做的是把文件上传、会话恢复和模型部署补齐

## 5. 当前阶段后的执行顺序

建议的实际开发顺序如下：

1. 校正 Harbor 文档中的旧架构表述
2. 部署并验证本地模型服务
3. 在 DeerFlow 中配置当前主模型
4. 接文件上传与 artifact 展示
5. 接历史会话和线程恢复
6. 补更完整的连接状态、错误状态和空状态
7. 补模型列表与模型切换入口
8. 做 MVP 打磨和验证
9. 补 Harbor 鉴权 / 业务后端代理层

## 6. 当前不建议立即做的事情

为了避免偏离 MVP 路线，当前不建议立即投入以下内容：

- 复杂技能系统
- 自动流水线执行
- UE Trace 专项解析
- 插件市场或插件框架
- 多模型调度系统
- 完整权限体系
- 大规模部署和集群化改造

## 7. 一句话总结

Harbor 从现在开始的正确推进方式是：先把桌面端直接接上 DeerFlow，把真实聊天链路跑通；之后再补上传、历史会话和状态展示，最后再在 DeerFlow 前面加 Harbor 自己的登录鉴权层。
