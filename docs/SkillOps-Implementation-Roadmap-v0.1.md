# SkillOps 实现路线图 v0.1

## 1. 文档目标

本文档用于定义 SkillOps 从当前阶段到 MVP 可运行版本的实际实现路线。它不是团队分工文档，而是项目的执行顺序说明，用于指导后续的设计、开发、联调和验证。

当前路线图以如下前提为基础：

- SkillOps 第一阶段采用桌面客户端形态
- 客户端需要支持本地工作目录接入
- 服务端采用 API Server + OpenClaw + Local Model Server 的基础架构
- 当前目标是先做出一个可运行的 MVP，而不是一次性实现完整平台能力

## 2. 总体原则

实现顺序遵循以下原则：

- 先确定边界，再开始编码
- 先搭骨架，再接能力
- 先跑通主链路，再做体验优化
- 先实现目录上下文接入，再讨论复杂本地任务执行
- 不提前实现后续才需要的复杂能力

## 3. 阶段划分

SkillOps 第一阶段建议分为 6 个阶段推进。

### 阶段 1：设计定稿

目标：把第一版产品范围和架构边界定清楚。

当前阶段需要完成：

- MVP 定义文档
- 系统架构设计文档
- 客户端架构设计文档
- 实现路线图文档

完成标准：

- 明确第一版做什么、不做什么
- 明确客户端、服务端、Agent 层、模型层的职责边界
- 明确客户端为何采用桌面应用

当前状态：

- 已基本完成

### 阶段 2：客户端骨架搭建

目标：搭建 SkillOps 桌面客户端基础工程，并做出可交互的产品壳。

本阶段需要实现：

- Electron + React + TypeScript + Vite 工程初始化
- 基础目录结构
- 主窗口创建
- 左侧会话栏 UI
- 主聊天区 UI
- 底部输入区 UI
- 基础深色主题和设计 token
- 本地目录选择能力接入
- 当前工作目录展示

本阶段暂不依赖真实服务端，可以先使用本地 mock 数据。

完成标准：

- 客户端能够启动并显示完整主界面
- 用户可以选择本地目录
- 用户可以在界面中看到当前工作目录状态
- 会话列表和消息区具备静态交互能力

交付物：

- 客户端可运行基础工程
- 第一版桌面 UI

### 阶段 3：服务端骨架搭建

目标：建立 SkillOps 服务端基础工程和数据层。

本阶段需要实现：

- FastAPI 工程初始化
- PostgreSQL 接入
- 基础配置管理
- 会话数据模型
- 消息数据模型
- 附件元数据模型
- 文件上传与下载接口
- 会话接口
- 消息接口

本阶段先不强依赖 OpenClaw，可先提供简单 mock 回复接口。

完成标准：

- 服务端工程可以启动
- 可以创建会话、写入消息、上传文件
- 客户端可通过接口读取历史会话和消息

交付物：

- 服务端基础工程
- 数据库表结构
- 基础 REST API

### 阶段 4：聊天链路打通

目标：打通客户端 -> API Server -> OpenClaw -> Model Server 的核心对话链路。

本阶段需要实现：

- OpenClaw 接入方案落地
- 本地模型服务接入
- 聊天请求转发
- 流式回复通道
- 对话结果持久化
- 基础错误处理和超时处理

完成标准：

- 用户可以在客户端发送消息并获得真实 Agent 回复
- 回复可以流式显示
- 会话和消息可以被保存和重新查看

交付物：

- 可用的真实聊天链路

### 阶段 5：MVP 能力补全

目标：补足第一版定义中的核心体验能力。

本阶段需要实现：

- 文件与图片上传接入实际聊天链路
- 历史会话切换与恢复
- 客户端连接状态展示
- 基础错误提示
- 附件展示卡片
- 空状态和加载状态
- 基础设置页或连接配置入口

完成标准：

- 用户可以完成一轮完整的真实使用流程
- 系统具备基础可用性和可演示性

交付物：

- SkillOps MVP 可运行版本

### 阶段 6：MVP 打磨与验证

目标：让 MVP 从“能跑”提升到“能稳定试用”。

本阶段需要实现：

- UI 细节打磨
- 日志和异常排查
- 局域网连接验证
- 文件上传边界验证
- 多轮会话验证
- 基础性能检查
- 打包与发布流程验证

完成标准：

- 团队内部可试用
- 主要链路稳定
- 具备最基本的演示和迭代能力

交付物：

- MVP 试用版本

## 4. 当前建议的实际起点

如果从现在正式进入实现，我建议立刻开始：

**阶段 2：客户端骨架搭建**

原因如下：

- 客户端形态已经确定
- UI 目标已经明确
- 本地目录能力是整个项目的关键前提
- 客户端壳搭起来后，后续联调会非常顺畅

也就是说，接下来最先动手的不是 OpenClaw，也不是 vLLM，而是：

- 初始化桌面客户端工程
- 搭出主界面
- 接入目录选择能力

## 5. 当前阶段后的执行顺序

建议的实际开发顺序如下：

1. 创建客户端工程
2. 完成客户端主界面布局
3. 接入本地目录选择
4. 增加本地会话假数据与基础交互
5. 创建服务端工程
6. 接入数据库和基础 API
7. 让客户端接入真实后端
8. 接入 OpenClaw 和模型服务
9. 补全附件、历史记录、状态展示
10. 做 MVP 打磨和验证

## 6. 当前不建议立即做的事情

为了避免偏离 MVP 路线，当前不建议立即投入以下内容：

- 复杂 Skill 系统
- 自动流水线执行
- UE Trace 专项解析
- 插件市场或插件框架
- 多模型调度系统
- 完整权限体系
- 大规模部署和集群化改造

这些都应该放到 MVP 跑通之后。

## 7. 对后续实现的要求

后续实现时，应始终遵守以下要求：

- 代码结构服务于后续扩展
- 配置不要写死
- 目录上下文能力先做最小可用版本
- UI 先追求稳和清晰，再追求复杂特效
- 所有实现都围绕主链路展开

## 8. 一句话总结

SkillOps 从现在开始的正确推进方式是：先把桌面客户端骨架做出来，再搭服务端基础能力，再接入 OpenClaw 和模型，把最小聊天主链路跑通，最后补全附件、历史记录和稳定性。
